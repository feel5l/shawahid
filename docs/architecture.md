# المعمارية الهندسية (Architecture)

يعتمد مشروع "ميثاق الأداء الوظيفي (زيد بن ثابت)" على معمارية حديثة تعتمد على فصل الواجهة الأمامية (Frontend) عن الواجهة الخلفية (Backend)، مع استخدام لغة **TypeScript** عبر كامل أجزاء النظام (Full-Stack Type Safety).

## 1. الهيكل العام للنظام

النظام مبني كـ **تطبيق صفحة واحدة (Single Page Application - SPA)** يتواصل مع خادم واجهة برمجة تطبيقات (RESTful API).

### الواجهة الأمامية (Client)
- **الإطار الأساسي:** React 18
- **أداة البناء (Bundler):** Vite (لضمان سرعة بناء وتطوير عالية).
- **إدارة المسارات (Routing):** `wouter` – تم اختياره لخفته وسرعته مقارنة بـ React Router، مما يقلل من حجم الحزمة النهائية.
- **إدارة الحالة وجلب البيانات:** `@tanstack/react-query` – يستخدم لجلب البيانات من الـ API، وتخزينها مؤقتًا (Caching)، وإعادة جلبها بذكاء عند الحاجة، وإدارة حالات التحميل والأخطاء.
- **التصميم وواجهة المستخدم:**
  - `Tailwind CSS`: لبناء واجهات متجاوبة (Responsive) تعتمد مبدأ Mobile-First.
  - `shadcn/ui`: مكتبة مكونات مبنية فوق Radix UI لتوفير مكونات يمكن الوصول إليها (Accessible) وسهلة التخصيص.

### الواجهة الخلفية (Server)
- **البيئة:** Node.js
- **إطار العمل:** Express.js
- **المصادقة:** أسلوب مبني على الجلسات (Sessions) باستخدام `express-session` و `passport.js`. 
- **التواصل مع قاعدة البيانات:** `drizzle-orm` (وهو ORM حديث يعتمد على TypeScript بالكامل ويتميز بأداء عالي).

## 2. هيكلة المجلدات (Directory Structure)

```text
├── client/                 # الجزء الخاص بالواجهة الأمامية
│   ├── src/
│   │   ├── components/     # مكونات React القابلة لإعادة الاستخدام (أزرار، نوافذ، بطاقات)
│   │   ├── hooks/          # Custom Hooks (مثل useAuth و useToast)
│   │   ├── lib/            # إعدادات المكتبات وتكوينها (مثل queryClient و utils)
│   │   └── pages/          # صفحات الموقع الرئيسية المرتبطة بنظام التوجيه
├── server/                 # الجزء الخاص بالواجهة الخلفية (API)
│   ├── index-dev.ts        # الملف الرئيسي لتشغيل الخادم في بيئة التطوير
│   ├── index-prod.ts       # الملف الرئيسي للإنتاج
│   ├── routes.ts           # تعريف مسارات الـ API ومنطق معالجة الطلبات
│   ├── replitAuth.ts       # إعدادات المصادقة (دخول المعلم، المدير، المنشئ)
│   └── storage.ts          # دوال التفاعل مع قاعدة البيانات
└── shared/                 # كود مشترك بين العميل والخادم
    └── schema.ts           # تعريف المخططات (Drizzle) ومخططات التحقق (Zod)
```

## 3. دورة حياة الطلب (Request Data Flow)

1. يقوم المستخدم بإجراء تفاعل واجهة مستخدم (مثل الضغط على زر "إضافة مؤشر").
2. يتم تشغيل الدالة (Mutation) عبر `react-query` داخل العميل.
3. يتم التحقق من صحة البيانات المُدخلة محلياً باستخدام مخططات `Zod` المشتركة (موجودة في مجلد `shared/`).
4. يتم إرسال طلب `fetch` أو `axios` (عبر دالة `apiRequest`) إلى الخادم. يشتمل الطلب على بيانات الاعتماد (Cookies/Sessions) `credentials: 'include'`.
5. يستقبل خادم Express الطلب. تمر العملية على Middleware للتحقق من الصلاحيات (مثل `isAuthenticated` أو `isPrincipal`).
6. إذا كان المستخدم مخولاً، يقوم الخادم باستخدام `storage.ts` لإجراء استعلام قاعدة البيانات عبر `drizzle-orm`.
7. تُرجع قاعدة البيانات (Neon PostgreSQL) النتائج إلى الخادم.
8. يعيد الخادم الرد كـ JSON، فيقوم `react-query` في الواجهة الأمامية بتحديث الـ Cache وإعادة رسم المكون (Re-render) لإظهار البيانات الجديدة.

## 4. القرارات الهندسية الرئيسية (Key Architectural Decisions)

- **استخدام Zod للتحقق المزدوج:** يتم استخدام نفس المخطط (Schema) للتحقق من النماذج (Forms) في الواجهة الأمامية، وللتحقق من صحة الحمولة (Payload) الواردة إلى مسارات الـ API في الواجهة الخلفية، مما يُلغي تكرار الكود ويضمن التطابق (Single Source of Truth).
- **دعم اللغة والاتجاه (RTL / Arabic):** التطبيق مصمم منذ اللحظة الأولى لدعم اتجاه (RTL) بشكل أصلي باستخدام خصائص Tailwind وسمات `dir="rtl"` في الهيكل الأساسي (`index.html` وتغليفة التطبيق).
- **التصميم المرن (Responsive Design):** تم الاعتماد على نظام الشبكات `grid` و `Flexbox` بشكل يضمن توافق النظام 100% مع أجهزة الجوال والشاشات الكبيرة بنفس الكفاءة.
